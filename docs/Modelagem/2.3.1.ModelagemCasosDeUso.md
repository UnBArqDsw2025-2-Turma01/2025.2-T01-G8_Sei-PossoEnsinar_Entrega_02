# 2.3.1 Diagrama de Casos de Uso

## Introdução

O diagrama de casos de uso descreve as interações entre os diferentes perfis de usuários e as funcionalidades disponíveis no sistema **Sei&PossoEnsinar**. Ele fornece uma visão clara de quem pode realizar determinadas ações, servindo como guia para o desenvolvimento e validação dos requisitos funcionais da aplicação.

No Sei&PossoEnsinar, o diagrama contempla ações de gerenciamento de perfil, solicitação de sessões, disponibilização de materiais, acompanhamento de calendário e interações entre alunos e monitores.

## Metodologia

O diagrama foi elaborado a partir dos requisitos levantados na etapa inicial de concepção do sistema e do protótipo de alta fidelidade. As funcionalidades foram organizadas de acordo com os papéis de **Usuário**, **Aluno** e **Monitor**. Foram utilizadas as relações `<<include>>` e `<<extend>>` da UML para indicar dependências obrigatórias e comportamentos condicionais.

A modelagem foi feita na ferramenta **Draw.io**, respeitando os padrões da UML.

## Especificações de Caso de Uso

#### UC1 – Realizar Cadastro

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que um novo `Usuário` crie uma conta no sistema, informando seus dados pessoais. |
| **Atores** | `Usuário` |
| **Pré-condição** | O `Usuário` não possui cadastro no sistema. |
| **Pós-condição** | O sistema registra a conta e o `Usuário` pode acessar via login. |
| **Fluxo Principal** | 1. O `Usuário` acessa a tela de cadastro.<br>2. O sistema exibe o formulário.<br>3. O `Usuário` insere os dados obrigatórios (nome, e-mail, senha).<br>4. O sistema valida os dados.<br>5. A conta é criada com sucesso. |
| **Fluxo Alternativo** | 3a. O `Usuário` insere informações adicionais, como competências ou descrição. |
| **Fluxo de Exceções** | 4a. Se o e-mail já estiver em uso ou a senha for inválida, o sistema informa erro.<br>5a. Em caso de falha no servidor, o sistema solicita que o `Usuário` tente novamente mais tarde. |

#### UC2 – Realizar Login

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que usuários previamente cadastrados acessem o sistema com suas credenciais. |
| **Atores** | `Usuário` |
| **Pré-condição** | O `Usuário` já deve possuir conta registrada. |
| **Pós-condição** | O sistema autentica o `Usuário` e libera acesso às funcionalidades. |
| **Fluxo Principal** | 1. O `Usuário` acessa a tela de login.<br>2. Insere e-mail e senha.<br>3. O sistema valida as credenciais.<br>4. O acesso é concedido. |
| **Fluxo Alternativo** | 3a. O `Usuário` pode redefinir a senha caso a tenha esquecido. |
| **Fluxo de Exceções** | 3b. Se as credenciais forem inválidas, o sistema exibe mensagem de erro.<br>4b. Em caso de falha técnica, o acesso não é concluído e o sistema orienta nova tentativa. |

#### UC3 – Gerenciar Perfil

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite ao `Usuário` alterar suas informações de perfil. |
| **Atores** | `Usuário`, `Aluno`, `Monitor` |
| **Pré-condição** | O `Usuário` deve estar autenticado. |
| **Pós-condição** | As alterações são salvas e exibidas no perfil. |
| **Fluxo Principal** | 1. O `Usuário` acessa o menu de perfil.<br>2. O sistema exibe as informações atuais.<br>3. O `Usuário` edita os campos desejados.<br>4. O sistema salva as alterações. |
| **Fluxo Alternativo** | 3a. O `Usuário` pode cancelar a edição antes de salvar. |
| **Fluxo de Exceções** | 4b. Se algum dado for inválido (ex.: formato de e-mail incorreto), o sistema não permite salvar.<br>4c. Em caso de falha técnica, o sistema informa erro e não conclui a alteração. |

#### UC4 – Gerenciar Competências

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Monitor` adicione, edite ou remova competências em seu perfil. |
| **Atores** | `Monitor` |
| **Pré-condição** | O `Monitor` deve estar autenticado e com perfil criado. |
| **Pós-condição** | As competências são atualizadas e exibidas no perfil. |
| **Fluxo Principal** | 1. O `Monitor` acessa “Gerenciar Competências”.<br>2. Insere ou altera as informações.<br>3. O sistema salva as mudanças.<br>4. O sistema confirma o sucesso da operação. |
| **Fluxo Alternativo** | 2a. O `Monitor` remove competências previamente cadastradas. |
| **Fluxo de Exceções** | 3a. Se as informações forem inválidas, o sistema impede a atualização.<br>3b. Em caso de falha no servidor, a alteração não é salva e o `Monitor` é informado. |

#### UC5 – Visualizar Calendário

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite ao `Usuário` acessar e visualizar o calendário de sessões agendadas. |
| **Atores** | `Usuário`, `Aluno`, `Monitor` |
| **Pré-condição** | O `Usuário` deve estar autenticado. |
| **Pós-condição** | O calendário é exibido com as sessões disponíveis ou já marcadas. |
| **Fluxo Principal** | 1. O `Usuário` acessa a opção de calendário.<br>2. O sistema carrega os eventos cadastrados.<br>3. O `Usuário` visualiza o calendário. |
| **Fluxo Alternativo** | 2a. O calendário pode ser filtrado por tipo de sessão ou disponibilidade de monitor. |
| **Fluxo de Exceções** | 2b. Se não houver eventos cadastrados, o sistema informa a ausência de sessões.<br>3b. Em caso de erro técnico, exibe mensagem de falha. |

#### UC6 – Solicitar Sessão

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Aluno` solicite uma sessão de acompanhamento com um `Monitor`. |
| **Atores** | `Aluno` |
| **Pré-condição** | O `Aluno` deve estar autenticado e ter encontrado um `Monitor`. |
| **Pós-condição** | A solicitação é registrada e aguarda resposta do `Monitor`. |
| **Fluxo Principal** | 1. O `Aluno` acessa “Solicitar Sessão” a partir do perfil do `Monitor`.<br>2. Escolhe o tema/habilidade da sessão.<br>3. O sistema registra a solicitação.<br>4. O sistema informa que a solicitação foi enviada. |
| **Fluxo Alternativo** | 2a. O `Aluno` pode incluir mensagem adicional para o `Monitor`. |
| **Fluxo de Exceções** | 3a. Se faltar informação obrigatória, o sistema não registra a solicitação.<br>3b. Em caso de falha técnica, exibe mensagem de erro. |

#### UC7 – Definir Horário da Sessão

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Monitor` defina os horários em que está disponível para sessões. |
| **Atores** | `Monitor` |
| **Pré-condição** | O `Monitor` deve estar autenticado. |
| **Pós-condição** | A disponibilidade do `Monitor` é atualizada e visível para os `Alunos`. |
| **Fluxo Principal** | 1. O `Monitor` acessa a seção de gerenciamento de agenda.<br>2. O sistema exibe o calendário do `Monitor`.<br>3. O `Monitor` seleciona os horários disponíveis.<br>4. O `Monitor` confirma a seleção.<br>5. O sistema salva os horários de disponibilidade. |
| **Fluxo Alternativo** | 3a. O `Monitor` pode definir a disponibilidade como recorrente (ex: toda terça-feira). |
| **Fluxo de Exceções** | 5b. Se ocorrer uma falha ao salvar, o sistema notifica o `Monitor` e pede para tentar novamente. |

#### UC8 – Tirar Dúvidas

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Aluno` e o `Monitor` se comuniquem por meio de um chat exclusivo para esclarecer dúvidas sobre a sessão agendada ou a habilidade. |
| **Atores** | `Aluno`, `Monitor` |
| **Pré-condição** | Uma sessão deve estar agendada ou uma conexão entre `Aluno` e `Monitor` estabelecida. |
| **Pós-condição** | As mensagens são trocadas e registradas no histórico do chat da sessão. |
| **Fluxo Principal** | 1. O `Aluno` acessa o chat da sessão agendada.<br>2. O `Aluno` escreve sua dúvida.<br>3. O `Aluno` envia a mensagem.<br>4. O sistema entrega a mensagem ao `Monitor`, que pode responder.<br>5. As interações são registradas. |
| **Fluxo Alternativo** | 4a. O `Monitor` pode encaminhar materiais complementares via chat. |
| **Fluxo de Exceções** | 3a. Se houver falha de conexão, a mensagem não é enviada e o sistema notifica.<br>5a. Em caso de falha técnica, o registro das interações pode não ser salvo. |

#### UC9 – Buscar Monitor Especializado

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Aluno` busque monitores com base em competências específicas. |
| **Atores** | `Aluno` |
| **Pré-condição** | O `Aluno` deve estar autenticado. |
| **Pós-condição** | O sistema exibe lista de monitores correspondentes ao filtro. |
| **Fluxo Principal** | 1. O `Aluno` acessa a busca.<br>2. Informa as competências desejadas.<br>3. O sistema exibe monitores compatíveis.<br>4. O `Aluno` escolhe um monitor. |
| **Fluxo Alternativo** | 2a. O `Aluno` pode aplicar múltiplos filtros (preço, avaliação, disponibilidade). |
| **Fluxo de Exceções** | 3a. Se não houver resultados, o sistema informa que não há monitores disponíveis.<br>3b. Em caso de falha técnica, exibe mensagem de erro. |

#### UC10 – Avaliar Sessões Concluídas

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Aluno` avalie uma sessão concluída com o `Monitor`. |
| **Atores** | `Aluno` |
| **Pré-condição** | O `Aluno` deve ter participado de uma sessão. |
| **Pós-condição** | A avaliação é registrada e vinculada ao `Monitor`. |
| **Fluxo Principal** | 1. O `Aluno` acessa “Avaliar Sessão” após a conclusão.<br>2. Informa nota e/ou comentário.<br>3. O sistema salva a avaliação. |
| **Fluxo Alternativo** | 2a. O `Aluno` pode optar por não deixar um comentário. |
| **Fluxo de Exceções** | 3a. Se os dados forem inválidos, o sistema impede salvar.<br>3b. Se houver falha técnica, a avaliação não é concluída. |

#### UC11 – Acessar Materiais

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Aluno` acesse materiais de apoio disponibilizados pelos `Monitores` para as sessões. |
| **Atores** | `Aluno` |
| **Pré-condição** | O `Aluno` deve estar autenticado e o `Monitor` deve ter disponibilizado materiais. |
| **Pós-condição** | O `Aluno` acessa os conteúdos anexados às sessões ou publicados pelos `Monitores`. |
| **Fluxo Principal** | 1. O `Aluno` acessa a área de materiais de uma sessão.<br>2. O sistema exibe os documentos.<br>3. O `Aluno` seleciona e abre o material. |
| **Fluxo Alternativo** | 2a. O `Aluno` pode filtrar os materiais por tema ou `Monitor`. |
| **Fluxo de Exceções** | 3a. Se o material não estiver disponível, o sistema informa erro.<br>3b. Se houver falha técnica, não será possível abrir o arquivo. |

#### UC12 – Gerenciar Disponibilidade

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Monitor` registre e atualize seus horários de disponibilidade. |
| **Atores** | `Monitor` |
| **Pré-condição** | O `Monitor` deve estar autenticado. |
| **Pós-condição** | Os horários ficam disponíveis para solicitação de sessões pelos `Alunos`. |
| **Fluxo Principal** | 1. O `Monitor` acessa “Gerenciar Disponibilidade”.<br>2. Informa ou atualiza os horários.<br>3. O sistema salva as alterações. |
| **Fluxo Alternativo** | 2a. O `Monitor` pode remover horários previamente cadastrados. |
| **Fluxo de Exceções** | 3a. Se houver falha no servidor, a alteração não é registrada e o sistema exibe mensagem de erro. |

#### UC13 – Atender Solicitações

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Monitor` visualize e atenda solicitações de sessão feitas por `Alunos`. |
| **Atores** | `Monitor` |
| **Pré-condição** | O `Monitor` deve ter recebido uma solicitação de sessão. |
| **Pós-condição** | A solicitação é respondida com aprovação, sugestão de horário ou recusa. |
| **Fluxo Principal** | 1. O `Monitor` acessa o painel de solicitações.<br>2. O sistema exibe as solicitações pendentes.<br>3. O `Monitor` analisa e responde (Aceita, Sugere Horário ou Recusa).<br>4. O sistema registra a resposta e notifica o `Aluno`. |
| **Fluxo Alternativo** | 3a. O `Monitor` pode enviar uma mensagem ao `Aluno` para tirar dúvidas antes de responder. |
| **Fluxo de Exceções** | 2a. Se não houver solicitações, o sistema informa ausência de registros.<br>4a. Em caso de falha técnica, a resposta não é registrada. |

#### UC14 – Disponibilizar Materiais

| Campo | Descrição |
| --- | --- |
| **Descrição** | Permite que o `Monitor` publique materiais de apoio para os `Alunos`. |
| **Atores** | `Monitor` |
| **Pré-condição** | O `Monitor` deve estar autenticado. |
| **Pós-condição** | O material fica acessível aos `Alunos`. |
| **Fluxo Principal** | 1. O `Monitor` acessa “Disponibilizar Materiais” em uma sessão ou em seu perfil.<br>2. Anexa documentos ou links.<br>3. O sistema valida e publica o conteúdo.<br>4. O sistema confirma sucesso. |
| **Fluxo Alternativo** | 2a. O `Monitor` adiciona descrição adicional ao material. |
| **Fluxo de Exceções** | 3a. Se o arquivo for inválido (tamanho, formato), o sistema não publica.<br>3b. Em caso de falha técnica, o envio não é concluído. |
